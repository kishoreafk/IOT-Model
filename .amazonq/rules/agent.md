This framework establishes the professional standard for spawning and coordinating
MULTIPLE subagents (minimum 5-10, scale up as needed) in parallel for EVERY task.

⚡ CRITICAL PRINCIPLE: SPAWN AGENTS FOR EVERY TASK ⚡
  NOT just major tasks. NOT just when "needed."
  Default action: Spawn agents FIRST, work SECOND.
  NO UPPER LIMIT on agent count — spawn more for complex work.

KEY PRINCIPLES:
  ✓ Agent swarms are your PRIMARY and DEFAULT tool
  ✓ Spawn MINIMUM 5-10 agents for every task, scale UP for complexity
  ✓ Large codebases? Spawn 15-20 agents. Massive? Spawn 25-30+
  ✓ Agents are FREE — more agents = FASTER and BETTER results
  ✓ No upper limit. Spawn as many as the task needs
  ✓ Parallel execution beats sequential work DRAMATICALLY
  ✓ More comprehensive coverage = fewer rework cycles
  ✓ Only skip agents for truly one-line, obvious tasks

RESULT: Consistently faster discovery, better decisions, higher quality outputs.

1.1 WHEN TO SPAWN AGENTS (ALWAYS — THIS IS THE DEFAULT)

*** SPAWN AGENTS BY DEFAULT FOR EVERY TASK ***
*** MINIMUM 5-10 AGENTS, SCALE UP FOR COMPLEXITY ***

Spawn minimum 5-10 agents in parallel for:

  ✓ PROJECT ANALYSIS & DISCOVERY (10 agents)
    → Exploring unfamiliar codebases
    → Understanding project architecture
    → Mapping dependencies and relationships
    → Analyzing configuration and setup patterns

  ✓ DATA INVESTIGATION (8-9 agents)
    → Analyzing dataset structure and formats
    → Researching data preprocessing pipelines
    → Examining data validation logic
    → Understanding data versioning and management

  ✓ MODEL & ARCHITECTURE RESEARCH (8-9 agents)
    → Investigating existing model implementations
    → Analyzing model configuration files
    → Researching checkpoint formats
    → Understanding model loading/saving patterns

  ✓ BUILD PHASE PREPARATION (8-10 agents per phase)
    → Before PHASE 1 (Scaffold) → Spawn 10 agents
    → Before PHASE 3 (Data Ingestion) → Spawn 9 agents
    → Before PHASE 5 (Model Architecture) → Spawn 9 agents
    → Before PHASE 6 (Training) → Spawn 8 agents

  ✓ TROUBLESHOOTING & DEBUGGING (10 agents)
    → When failures occur — ALWAYS spawn agent swarm
    → When root cause is unclear
    → When investigating performance issues
    → When analyzing error patterns

  ✓ DOCUMENTATION & RESEARCH (6-8 agents)
    → Reading comprehensive documentation
    → Researching best practices
    → Understanding framework patterns
    → Learning from existing implementations

  ✓ INTEGRATION & VALIDATION (7-8 agents)
    → Validating system integration points
    → Checking compatibility between components
    → Analyzing deployment configurations
    → Reviewing monitoring and logging setup

  ✓ CODE REVIEW & ANALYSIS (7-9 agents)
    → Analyzing code quality and patterns
    → Reviewing implementation approaches
    → Checking for bugs or improvements
    → Understanding complex logic

  ✓ REFACTORING & OPTIMIZATION (7-8 agents)
    → Planning refactoring strategy
    → Analyzing performance bottlenecks
    → Researching optimization techniques
    → Validating refactoring approach

  ✓ FEATURE ADDITION & EXPANSION (8-9 agents)
    → Understanding existing feature implementations
    → Planning new feature architecture
    → Researching similar implementations
    → Identifying integration points for new features

  ✓ EVEN SMALL TASKS (5-6 agents)
    → Reading a single file? Spawn 5 agents with different analysis angles
    → Implementing a utility function? Spawn agents to validate approach
    → Making a small fix? Spawn agents to check for related issues
    → Writing documentation? Spawn agents to review for clarity and completeness

1.2 HOW MANY AGENTS TO SPAWN

SPAWN COUNT BY TASK COMPLEXITY (NO UPPER LIMIT):

  5-6 agents: BARE MINIMUM for simple, narrow tasks
    → Single file review with one clear angle
    → Quick documentation check (limited scope)
    → Simple configuration verification
    → Single-function utility review

  7-8 agents: STANDARD baseline for most tasks
    → Framework exploration (2-3 modules)
    → Multi-component analysis
    → Cross-module dependency investigation
    → Code review and refactoring
    → Medium-complexity bug investigation

  9-10 agents: RECOMMENDED for typical significant work
    → Full module or subsystem analysis
    → Complex architecture discovery
    → Comprehensive pre-build research (single domain)
    → Multi-phase preparation
    → Major failures and debugging
    → Strategic planning for medium-large features

  11-15 agents: FOR LARGE, MULTI-DOMAIN ANALYSIS
    → Entire codebase exploration (large projects)
    → Distributed system architecture review
    → End-to-end pipeline investigation
    → Multiple independent subsystems
    → Cross-team component analysis
    → Major refactoring or migration planning

  16-20 agents: FOR ENTERPRISE-SCALE WORK
    → Massive codebases (100k+ lines)
    → Multiple interconnected services
    → Deep infrastructure investigation
    → Organization-wide architectural review
    → Complete migration or platform rebuild

  20+ agents: FOR EXHAUSTIVE, MULTI-PERSPECTIVE ANALYSIS
    → Extremely complex systems (1M+ lines of code)
    → Novel/unfamiliar tech stack deep dives
    → Complete architectural/strategic decisions
    → High-stakes system redesign
    → When uncertainty is high = spawn MORE agents

  RULE OF THUMB: 
    • Minimum = 5-6 agents (only for trivial tasks)
    • Standard = 8-10 agents (default for most work)
    • Recommended = 10-15 agents (when in doubt, go here)
    • Optional escalation = 15-20+ agents (for complex/large work)
    • When uncertain about complexity = spawn MORE, not fewer
    • NO UPPER LIMIT — agents are parallel and fast

2.1 PATTERN A: FULL PROJECT DISCOVERY (10-20 AGENTS)

Use when entering a NEW project or unfamiliar codebase.
Scaling guidance: Small project (10 agents) → Medium (15 agents) → Large (20+ agents)

AGENT ASSIGNMENTS:
  
  Agent 1: PROJECT STRUCTURE MAPPER
    Task: "List complete directory structure, identify main modules,
           map relationships between folders. Report on organization pattern."
    Expected: Folder hierarchy, module count, structural patterns

  Agent 2: CONFIGURATION INVESTIGATOR
    Task: "Find ALL config files (YAML, JSON, TOML, INI). List contents,
           identify what each configures, report on config patterns."
    Expected: Config file inventory, structure, key settings

  Agent 3: DATA PIPELINE ANALYST
    Task: "Analyze all data handling code. Map data flow, identify loaders,
           preprocessors, validators. Report on data architecture."
    Expected: Data flow diagram, loader implementations, validation logic

  Agent 4: MODEL ARCHITECTURE RESEARCHER
    Task: "Examine all model definitions, architectures, pretrained weights.
           Identify model patterns, dependencies, checkpoint handling."
    Expected: Model inventory, architecture patterns, checkpoint formats

  Agent 5: TRAINING SYSTEM INVESTIGATOR
    Task: "Analyze training scripts, loops, schedulers, optimizers.
           Map training pipeline, identify config relationships."
    Expected: Training flow, optimizer patterns, scheduler configurations

  Agent 6: EVALUATION & METRICS ANALYST
    Task: "Research all metrics, evaluation scripts, benchmark patterns.
           Identify evaluation methodology and reporting."
    Expected: Metrics inventory, evaluation patterns, reporting mechanisms

  Agent 7: DEPLOYMENT INFRASTRUCTURE EXPLORER
    Task: "Examine serving code, APIs, Docker, deployment configs.
           Map deployment pipeline and inference paths."
    Expected: Serving architecture, API endpoints, deployment patterns

  Agent 8: TEST & VALIDATION FRAMEWORK ANALYZER
    Task: "Research all tests, validation, CI/CD. Identify testing patterns
           and quality assurance mechanisms."
    Expected: Test inventory, CI/CD workflows, quality gates

  Agent 9: DEPENDENCIES & REQUIREMENTS AUDITOR
    Task: "Analyze all dependencies, requirements.txt, package versions.
           Identify version constraints and compatibility patterns."
    Expected: Dependency tree, version constraints, compatibility notes

  Agent 10: DOCUMENTATION RESEARCHER
    Task: "Review all documentation, READMEs, comments, docstrings.
           Summarize project patterns, conventions, best practices."
    Expected: Documentation summary, patterns, conventions

CONSOLIDATION:
  Combine all 10 summaries into:
  → Holistic project understanding
  → Architecture blueprint
  → Key decision points
  → Risk and dependency analysis
  → Preparation for building or modifying

2.2 PATTERN B: BEFORE-BUILD INVESTIGATION (9-15 AGENTS)

Use before starting PHASE 1, 3, 5, or 6 of ML pipeline build.
Start with 9 agents → add more if task complexity warrants

AGENT ASSIGNMENTS:

  Agent 1: EXISTING SCAFFOLD PATTERNS
    Task: "Review current project structure. What patterns are established?
           What folder organization is used? Report standards observed."

  Agent 2: CONFIG SYSTEM ANALYSIS
    Task: "Analyze existing config files and loading patterns.
           How are configs structured? What tools (Pydantic, DVC)?"

  Agent 3: DATA HANDLING PATTERNS
    Task: "Examine how data is currently loaded, split, validated.
           What libraries? What patterns? Document findings."

  Agent 4: MODEL LOADING PATTERNS
    Task: "Analyze model loading, checkpoint management, weight handling.
           What formats? What best practices? Report patterns."

  Agent 5: TRAINING LOOP PATTERNS
    Task: "Research existing training loops, schedulers, early stopping.
           What patterns are established? What conventions?"

  Agent 6: METRICS & EVALUATION SETUP
    Task: "Analyze current metrics, evaluation approaches, reporting.
           What metrics are measured? How is performance tracked?"

  Agent 7: EXPORT & SERVING PATTERNS
    Task: "Examine model export, inference serving, API patterns.
           What formats supported? API structure?"

  Agent 8: TESTING & CI/CD SETUP
    Task: "Research test framework, GitHub Actions, build pipeline.
           What testing patterns? What quality gates?"

  Agent 9: MONITORING & LOGGING PATTERNS
    Task: "Analyze logging setup, monitoring infrastructure, alerting.
           What logging framework? What metrics tracked?"

CONSOLIDATION:
  Create informed blueprint that:
  → Aligns with existing patterns
  → Reuses established conventions
  → Avoids redundancy
  → Maintains consistency
  → Identifies integration points

2.3 PATTERN C: DEBUGGING & TROUBLESHOOTING (10-20+ AGENTS)

Use when multiple failures occur or root cause is unclear.
Start with 10 agents → escalate to 15-20+ if failures are complex/cascading

AGENT ASSIGNMENTS:

  Agent 1: DATA PIPELINE DEBUGGER
    Task: "Trace data loading failures. Check data formats, loaders,
           validation. Identify data-related issues."

  Agent 2: CONFIG VALIDATION DEBUGGER
    Task: "Analyze config loading errors. Check validation, types,
           required fields. Investigate config issues."

  Agent 3: MODEL LOADING DEBUGGER
    Task: "Trace model loading failures. Check architectures, weights,
           compatibility. Report model loading issues."

  Agent 4: TRAINING FAILURE ANALYZER
    Task: "Investigate training loop errors. Check loss calculations,
           optimizer updates, convergence issues."

  Agent 5: EVALUATION METRICS DEBUGGER
    Task: "Analyze evaluation failures. Check metric calculations,
           mismatches, reporting errors."

  Agent 6: CHECKPOINT FAILURE INVESTIGATOR
    Task: "Debug save/load failures. Check pickle compatibility,
           state dict mismatches, checkpoint format issues."

  Agent 7: EXPORT FORMAT DEBUGGER
    Task: "Trace export errors. Check ONNX, TorchScript, TFLite issues.
           Investigate format conversion problems."

  Agent 8: API & INFERENCE DEBUGGER
    Task: "Debug API endpoint failures. Check request/response handling,
           batch processing, serialization issues."

  Agent 9: CONTAINERIZATION DEBUGGER
    Task: "Analyze Docker build/run failures. Check Dockerfile,
           dependencies, runtime environment."

  Agent 10: TEST & CI/CD DEBUGGER
    Task: "Investigate test failures. Check test logic, assertions,
           CI/CD pipeline errors."

CONSOLIDATION:
  → Identify root causes across all failure points
  → Prioritize fixes by impact
  → Create unified recovery plan
  → Prevent cascading failures

3.1 AGENT PROMPT STRUCTURE (PROFESSIONAL)

Every agent prompt MUST include:

  ✓ CLEAR OBJECTIVE
    "Your task is to <specific action>..."

  ✓ SCOPE DEFINITION
    "Focus on <domain/component>. Ignore <out-of-scope items>."

  ✓ INVESTIGATION DEPTH
    "Provide: <bullet points of expected findings>"

  ✓ OUTPUT FORMAT
    "Return a summary with: <structure of findings>"

  ✓ THOROUGHNESS LEVEL
    "Use thoroughness: quick | medium | thorough"

3.2 EXAMPLE AGENT PROMPTS

GOOD PROMPT (Specific)
  "Explore the data pipeline in src/ingestion/. List all DatasetClasses,
   document their __init__ and getitem methods. Analyze how data is loaded,
   cached, and validated. Report findings: [1] Class inventory, [2] Data
   flow per class, [3] Caching strategy, [4] Validation logic.
   Thoroughness: thorough"

BAD PROMPT (Vague)
  "Look at the data code."

GOOD PROMPT (Clear scope)
  "Investigate config handling in configs/. Find all config files and
   loaders, document structure, identify Pydantic models used.
   Report: [1] Config inventory, [2] Loader patterns, [3] Validation rules."

BAD PROMPT (Unclear)
  "Check configs please."

  Step 3: CREATE unified understanding/architecture map
  Step 4: HIGHLIGHT decision points and unknowns
  Step 5: DOCUMENT consolidated findings in clear format
  Step 6: PROCEED with informed next steps

6.1 MANDATORY AGENT RULES (FOR ALL TASKS)

  ⚡ RULE 1: Default to spawning 5-10 agents for EVERY task
    → Not "optional" or "nice-to-have"
    → MANDATORY for all non-trivial work
    → DEFAULT action, not exception
    → Spawn FIRST, work SECOND

  ⚡ RULE 2: Spawn agents BEFORE doing anything else
    → Before writing code
    → Before making decisions
    → Before reading files yourself
    → NEVER proceed without agent insights

  ⚡ RULE 3: Wait for ALL agent results
    → Read complete summaries
    → Don't skip any agent
    → Don't proceed on partial findings
    → Consolidate EVERYTHING

  ⚡ RULE 4: Parallel > Sequential ALWAYS
    → Spawn all agents simultaneously
    → Never chain agents sequentially
    → Parallel execution = faster results
    → Each agent independent and complete

  ⚡ RULE 5: Be VERY specific with agent prompts
    → Crystal-clear objectives
    → Well-defined scope
    → Expected outputs detailed
    → Appropriate thoroughness level
    → No vague or ambiguous language

  ⚡ RULE 6: Consolidate findings BEFORE ANY action
    → Don't act on partial information
    → Create unified understanding from all agents
    → Document integrated findings
    → Use consolidated insights for decisions

  ⚡ RULE 7: Apply agent.md + new.md together
    → agent.md = HOW to spawn and organize agents
    → new.md = WHAT to build (RULE 0 integrates this)
    → Both are RULES for the same LLM
    → They work together seamlessly

6.2 ANTI-PATTERNS (THINGS NOT TO DO)

  ✗ Don't code first, then investigate
    → Investigation should happen first
    → Decisions must be informed

  ✗ Don't chain agents sequentially
    → Defeats the purpose of parallelism
    → Wastes time with serial investigation

  ✗ Don't spawn 1-2 agents
    → Too narrow perspective
    → Misses interdependencies
    → Increases rework

  ✗ Don't use vague agent prompts
    → "Look at the code"
    → "Check things"
    → Be specific and clear

  ✗ Don't ignore agent findings
    → Use discoveries to inform decisions
    → Align implementation with patterns found
    → Leverage agent intelligence

  ✗ Don't proceed without consolidation
    → Don't act on partial findings
    → Wait for complete picture
    → Then make informed decisions

Q1: Is spawning 10+ agents overkill?
A: No. Parallel agents finish faster than sequential work. Each agent is
   cheap and fast. 10-20+ agents in parallel = one comprehensive discovery pass.
   More agents = FASTER results with less rework. Never hesitate to spawn more.

Q2: What if agents have overlapping tasks?
A: Slight overlap is GOOD. Redundancy reveals consistency. If agents find
   different things in overlapping areas, you've found an inconsistency
   worth investigating.

Q3: How many agents for simple vs complex tasks?
A: Minimum 5-6 agents for simple tasks. Use 10+ for typical work. Escalate
   to 15-20+ for large/complex codebases. Default to HIGHER counts when in doubt.
   Agents are parallel and fast — more coverage = faster discovery = better results.

Q4: How long to wait for agents?
A: Wait for ALL agents. Don't rush. Agent execution is parallel, so
   total time ≈ slowest agent. Waiting is faster than re-doing work.

Q5: Can I modify agent tasks mid-investigation?
A: No. Define tasks clearly before spawning. Changing tasks mid-run
   creates confusion. Spawn new agents if pivot needed.

Q6: What if an agent fails to find anything?
A: That's data. "Nothing found" is still a finding. Document it and
   understand why. It might mean the code area is empty, or the approach
   was wrong.

Q7: Should I always consolidate findings?
A: YES. Always. Create a unified summary. Don't proceed on partial
   agent results. Consolidation is non-optional.
